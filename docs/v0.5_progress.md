# v0.5 途中経過 — 設計修正と軌道修正 (2026-07-07)

## 現状サマリ

v0.5 Voice Input のコンポーネント実装は完了。残りは繋ぎこみ（#114）と Unity UI（#115）。

コードレビューにより以下の問題が発見された。これらを #114 着手前に修正する必要がある。

## 1. アーキテクチャ修正: STT の担当を Go に変更

### 背景
- 設計 doc では STT を Python の責務としていた
- 実際のコードは Go と Python の両方に STT クライアントが存在
- 「外部API通信の集約」は Go Runtime の責務であり、faster-whisper は外部サービス
- Go の STT クライアント（`internal/stt/client.go`）の方が context.Context 対応・エラーハンドリングが優れている

### 決定
STT は Go Runtime が担当する。Python AI Service は VAD + LLM に専念。

### 修正後の v0.5 データフロー

```
Unity (WinPC)
  ↓ WebSocket binary: audio_chunk (100ms PCM)
Go Runtime (X1C6)
  ↓ HTTP: POST /vad/chunk (PCM octet-stream)
Python AI Service (X1C6)
  ↓ Silero VAD → speech_start / speech_end 検出
  ↓ speech_end 時: WAV (base64) をレスポンスに含める
Go Runtime
  ↓ Go STT client → WinPC faster-whisper /v1/transcribe
  ↓ text
Go Runtime
  ↓ handleTextMessageAgent (LLM → TTS)
  ↓ WebSocket: ai_response (audio + text)
Unity
```

### 修正対象ファイル
- `docs/architecture.md` v0.5 セクション: Python の責務から STT を削除、Go に移動
- `src/local_ai_companion/stt.py`: 削除（Go 側で担当）
- `tests/test_stt.py`: 削除（Python 側 STT テスト不要に）

## 2. STT multipart フィールド名の統一

### 問題
WinPC faster-whisper サーバーは FastAPI の `UploadFile` で `file` を受け取る。
Go STT クライアントは `audio_file` で送信しているため通信できない。

### 修正
- `internal/stt/client.go`: `CreateFormFile("audio_file", ...)` → `CreateFormFile("file", ...)`

---

## 3. Python VAD が WAV を返す仕様の確認

### 現状
`vad.py` の `process_chunk` は speech_end 時に `{"event": "speech_end", "audio_wav": "<base64 WAV>"}` を返す。
Go がこれを受け取って STT に渡す設計になっている。

### 確認結果
✅ この動作は正しい。修正不要。

---

## 4. STT レスポンスに duration フィールドを追加

### 問題
API 契約は `{"text": "...", "duration": 3.2, "error": null}` を定義しているが、
Go STT client の `Result` 構造体に `duration` がない。

### 修正
- `internal/stt/client.go`: `Result` に `Duration float64` を追加、faster-whisper レスポンスから `duration` をパース

---

## 5. API 契約ドキュメントの修正

### 問題点
- VAD エラーレスポンス形式: 契約 `{"error": "..."}` → 実装 `{"error": {"code": "...", "message": "..."}}`
- STT 実装箇所: 契約に `vad_server.py` とあるが実際は `server.py`
- STT エンドポイントの記述を「Python → WinPC」から「Go → WinPC」に更新

### 修正
- `docs/api_contracts.md` の上記項目を実装に合わせて更新

## 正しい #114 実装フロー（修正後）

```
handleAudioChunk(data []byte):
  1. ParseAudioChunk → AudioChunk
  2. POST /vad/chunk (PCM octet-stream)
  3. レスポンス {"event": "speech_start"} → WebSocket broadcast vad_event
  4. レスポンス {"event": "speech_end", "audio_wav": "..."}
     → base64 decode WAV
     → stt.Client.Transcribe(ctx, wavData, 16000, "ja")
     → テキスト取得
     → handleTextMessageAgent に投入
  5. StateMachine: LISTENING (VAD中) → THINKING (STT+LLM中) → SPEAKING → IDLE
```
