# ローカルAIアシスタント開発方針書

## 目的

ローカル環境で動作するAIアシスタントを開発する。  
最終的にはGateboxのように、音声で会話でき、キャラクター表示を持ち、ユーザーの作業環境や生活情報と連携する常駐型アシスタントを目指す。

ただし、開発は薄いプロトタイプを全体に広げるのではなく、**各段階を完成度高く仕上げてから次の段階へ進む**方針とする。

---

## 開発思想

### 基本方針

本プロジェクトでは、以下の方針を重視する。

```text
全体を10%ずつ作るのではなく、
1ステップを可能な限り100%に近づけてから次へ進む。
```

つまり、以下のような作り方は避ける。

```text
STTも微妙
LLMも微妙
TTSも微妙
キャラ表示も微妙
記憶も微妙
結果として全体が微妙
```

代わりに、各段階で単体の成果物として成立する状態を目指す。

---

## 開発順序

最終的な開発順序は以下とする。

```text
1. 会話コア
2. 音声I/O
3. キャラクター表示
4. 記憶・RAG
5. 自律行動
```

以前の分類に当てはめると、

```text
2 → 3 → 1 → 4 → 5
```

となる。

---

## Phase 1: 会話コア

### 目的

テキスト入力に対して、LLMが安定した応答を返す会話エンジンを作る。  
この段階では音声入力、音声出力、キャラクター表示は扱わない。

### 入力

```text
ユーザーのテキスト
```

### 出力

LLMは単なる自然文ではなく、後続システムで扱いやすい構造化データを返す。

例：

```json
{
  "text": "それはかなり良い判断ですね。",
  "emotion": "thinking",
  "motion": "nod",
  "speak_style": "normal",
  "interruptible": true
}
```

### 要件

- ローカルLLMとクラウドLLMを切り替え可能にする
- 返答形式をJSONで安定させる
- JSONが壊れた場合の復旧処理を持つ
- キャラクター口調を安定させる
- 返答が長くなりすぎないよう制御する
- 会話履歴を管理する
- エラー時のフォールバック応答を用意する
- ログを保存する
- 後続の音声出力・キャラ表示に接続できる形式にする

### 完成条件

以下を満たしたらPhase 1完了とする。

```text
・テキスト入力で安定して会話できる
・返答が常に所定のJSON形式で得られる
・モデルを差し替えてもアプリ側の仕様が崩れない
・会話ログが残る
・異常応答時に安全に復旧できる
```

---

## Phase 2: 音声I/O

### 目的

Phase 1の会話コアに音声入力と音声出力を接続し、音声AIアシスタントとして成立させる。

この段階でもキャラクター表示はまだ扱わない。

### 構成

```text
マイク
↓
音声区間検出
↓
STT
↓
会話コア
↓
TTS
↓
スピーカー
```

### 要件

#### 音声入力

- 話し始めを検出できる
- 話し終わりを検出できる
- 環境音による誤爆を抑える
- 認識結果をテキストで確認できる
- 認識ミス時にキャンセルできる
- マイクデバイス変更に対応する

#### 音声出力

- TTSで自然に読み上げる
- 音声再生キューを管理する
- 再生中断ができる
- 発話中に自分のTTS音声を再入力しない
- 音声出力とテキスト出力を同期できる

### 完成条件

```text
・声で話しかけると返事が返ってくる
・会話のテンポが許容範囲に収まる
・TTSの音声を自分で拾ってループしない
・割り込みやキャンセルができる
・音声入力が実用レベルで安定する
```

### 重要ポイント

Gatebox系の体験で最も重要なのは、LLMの賢さだけではない。

```text
聞き取りの自然さ
返答までのテンポ
発話の中断しやすさ
誤認識時のリカバリ
```

ここが弱いと、後でキャラクター表示を追加しても高品質な置物になる。

---

## Phase 3: キャラクター表示

### 目的

Phase 1・2で完成した会話システムに、キャラクターの身体を追加する。

この段階で初めて、Live2D、VRM、Unity、VTube Studioなどを検討する。

### 入力

Phase 1で定義した応答JSONを利用する。

```json
{
  "text": "なるほど、それは面白いですね。",
  "emotion": "happy",
  "motion": "nod",
  "speak_style": "normal",
  "interruptible": true
}
```

### 出力

```text
キャラクター表示
表情変更
モーション再生
口パク
字幕表示
待機モーション
```

### 要件

- `emotion` に応じて表情を変更する
- `motion` に応じて動作を切り替える
- TTS音声に合わせて口パクする
- 発話中と待機中の状態を分ける
- 字幕を表示する
- サブモニターなどに常駐できる
- 長時間起動しても落ちにくい
- 落ちた場合に復帰しやすい

### 完成条件

```text
・AIの発話に合わせてキャラクターが自然に動く
・音声、字幕、口パクが破綻しない
・話していない時も自然に待機する
・外部APIから発話と表情を制御できる
```

### 注意点

キャラ表示を最初に作らない理由は、見た目の実装が本質を飲み込みやすいからである。

```text
Live2D調整
VRM調整
Unity実装
表情制御
モーション制御
```

これらは楽しいが、先にやると会話体験の完成が遅れる。  
まず音声AIとして成立させ、その後にキャラクターを載せる。

---

## Phase 4: 記憶・RAG

### 目的

ユーザー固有の情報を扱えるようにし、単なる会話AIから個人用アシスタントへ発展させる。

### 扱う情報

```text
会話ログ
作業メモ
PC構成
GPU情報
過去のトラブル記録
予定
ローカルファイル
技術メモ
```

### 要件

- 記憶する情報を明示的に管理できる
- 古い情報と新しい情報を区別できる
- 必要な情報だけ検索できる
- 根拠を提示できる
- 忘れさせる操作ができる
- 個人情報を不用意に読み上げない
- 会話に自然に検索結果を混ぜられる

### 完成条件

```text
・過去の会話や作業記録を参照できる
・PC構成やトラブル履歴を覚えている
・根拠付きで回答できる
・不要な記憶を削除できる
```

### 例

```text
ユーザー: 前にV100のファンノイズで何話してたっけ？
AI: 高音が気になるという話でした。対策として、風量よりもファンの種類と取り付け方法を見直す方向で考えていました。
```

---

## Phase 5: 自律行動

### 目的

ユーザーから話しかけられた時だけでなく、必要に応じてAI側から働きかける常駐アシスタントにする。

### 例

```text
GPU温度監視
作業ログ要約
予定通知
寝る時間の通知
長時間作業の警告
エラー検知
定期的な状態報告
```

### 要件

- 通知条件を明確に定義する
- 不要なタイミングで話しかけない
- ユーザーが一時停止できる
- 重要度に応じて通知方法を変える
- ログを残す
- 誤通知を減らす

### 完成条件

```text
・必要な時だけ声をかける
・常駐していて邪魔にならない
・PCや生活の状態を見て支援できる
・通知条件をユーザーが調整できる
```

## 全体アーキテクチャ

### 最終形

```text
[User]
  ↓ voice/text
[Input Layer]
  ├─ Text Input
  └─ STT

  ↓

[Conversation Core]
  ├─ LLM
  ├─ Prompt Manager
  ├─ History Manager
  └─ Response Formatter

  ↓ JSON

[Output Layer]
  ├─ TTS
  ├─ Subtitle
  ├─ Character Controller
  └─ Motion/Expression Controller

  ↓

[Memory Layer]
  ├─ Conversation Logs
  ├─ Notes
  ├─ PC State
  └─ RAG Search

  ↓

[Autonomous Layer]
  ├─ Scheduler
  ├─ Monitor
  └─ Notification Controller
```

## 技術選定の考え方

### LLM

最初は差し替え可能にする。  
ローカルLLM固定にしない。

```text
ローカルLLM
クラウドLLM
OpenAI互換API
Ollama系
llama.cpp系
```

を切り替えられるようにする。

### STT

音声認識は後から差し替えられるようにする。

```text
Whisper系
faster-whisper
外部API
OS標準音声認識
```

### TTS

最初は扱いやすいものを使う。

```text
VOICEVOX
Piper
Style-Bert-VITS系
外部TTS API
```

### キャラクター表示

段階的に選ぶ。

```text
最初: なし
次: 簡易UI
次: Live2D / VTube Studio
次: VRM / Unity
```

## 開発上の優先順位

### 優先度 高

```text
会話テンポ
安定性
ログ
エラー復旧
音声入出力の快適さ
仕様の分離
```

### 優先度 中

```text
キャラの見た目
表情の豊かさ
口調の細かい調整
記憶機能
```

### 優先度 低

```text
高度な自律行動
凝った3D演出
複雑な感情モデル
完全な人格設定
```

## やらないこと

初期段階では以下をやらない。

```text
最初からLive2D/VRMに全力投入しない
最初から自律行動を入れない
最初から長期記憶を入れない
最初から完璧な人格設定を作らない
全部を同時に繋げない
```

---

## 最初に作るべきもの

最初の成果物はこれ。

```text
テキスト入力に対して、
LLMが安定したJSON応答を返す会話コア
```

最低限のI/Oは以下。

```text
Input:
  ユーザーのテキスト

Output:
  text
  emotion
  motion
  speak_style
  interruptible
```

これができれば、後から音声出力、キャラクター表示、記憶、自律行動を安全に追加できる。

---

## 結論

このプロジェクトでは、最初からGateboxの見た目を作らない。  
まずは**会話の核**を完成させる。

開発順序は以下。

```text
会話コア
↓
音声I/O
↓
キャラクター表示
↓
記憶・RAG
↓
自律行動
```

この順番にすることで、各段階が単体で成立し、後続機能に流用できる資産になる。

最終的に目指すのは、単なるローカルLLMではなく、

```text
ユーザーのPC環境・作業・生活に紐づいた常駐AIキャラクター
```

である。
